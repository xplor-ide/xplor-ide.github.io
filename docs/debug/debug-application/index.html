Debug Application

Tags:
  • Debug

XploR Eos IDE provides an integrated Debug View, where users can start, stop, and control the debugging process, view variables, and monitor the call stack. Additionally, multiple data views give a comprehensive overview by revealing the status of different subsystems of the target. This section outlines some of these views and features.

The debug behavior will differ depending on whether an Automatic or Manual debug launch is performed.

For an Automatic debug launch, you will switch from the Explorer to the Debug view in the Side Bar, displaying the default arrangement of views as shown below. The Call Stack view provides information on the current program. In the source code view, the program is usually displayed as halted at the beginning of the main() function (though this may vary depending on the loaded program). Below the Threads, Call Stack View, the Variables, Watch, Breakpoint are also available.

The following sections showcase some of the most commonly used views specifically designed for debugging.

Debug View

The basic debugging controls are located in the top-left corner of the Debug View, with toolbars providing various functionalities:

Top Toolbar:

  • Start Debugging: Initiates a debug session using the selected launch configuration.
  • Launch Configuration: A dropdown menu listing available launch configurations.
  • Open launch.json: Opens the JSON file containing the launch configuration settings.
  • Debug Console: Opens the Debug Console view for interacting with the debugger.

Bottom Toolbar

  • Continue: Resumes the execution of the target core.
  • Pause: Stops the target core’s execution. The Continue button changes to Pause when clicked, and vice versa.
  • Step Over: Executes the current source line and skips over any functions or subroutines, running their internal code in one step.
  • Step Into: Executes the current line but steps inside functions or subroutines, allowing for step-by-step debugging of the internal code.
  • Step Out: Completes the execution of the current function or subroutine and returns to the calling location.
  • Restart: Restarts the session when symbols or a program have been loaded, resetting.
  • Stop: Ends the debug session and disconnects from hardware or QEMU.

Threads View

The Threads View is located at the top of the Debug View on the left side. It shows a list of available cores for debugging, along with the status of each core.

Right-clicking on a core will bring up the following options:

  • Continue: Resumes execution of the target core.
  • Step Over: Executes the next line of code, skipping over the details of subroutines or functions and running them as a whole.
  • Step Into: Executes the next line of code but goes inside subroutines or functions, allowing you to step through the internal code line by line.
  • Step Out: Completes the remaining execution of the current subroutine and returns to the point where it was originally called.

Call Stack View

The Call Stack View displays the functions currently on the stack, listed in the order in which they are being executed.

Each entry displays the following details:

  • The name of the function being executed
  • The memory address of the function
  • The name of the file containing the function
  • The line number in the file where the function is defined

By right-clicking in the Call Stack View, you can access options to copy the call stack, restart frame or open the Disassembly View.

Variables View

The Variables view displays the local variables associated with the function currently highlighted in the Call Stack View. This includes:

  • Local Variables: Variables defined within the currently active function
  • Static Variables: Variables that maintain their value across multiple calls to the same function.
  • Global Variables: Variables that can be accessed from any part of the program.

Additionally, the view includes the CPU register that shows details in the Register View

Watch View

Watch view allows developers to monitor specific variables or expressions during debugging. This feature is crucial for tracking values as they change throughout program execution, enabling efficient troubleshooting.

Register View

The Register View is a subsection of the Variables View, providing information about CPU registers during a debugging session. It shows a list of available CPU registers, typically including general-purpose registers, special registers, and status registers.

Common registers displayed may include:

  • General-Purpose Registers (e.g., R0, R1, R2): Used for arithmetic operations and data storage.
  • Status Registers (e.g., PSW, SR): Contain flags that provide information about the processor’s state.
  • Pointer Registers (e.g., SP, PC): Used to store memory addresses for the stack pointer and program counter.

The Register View updates in real-time as the program executes, allowing developers to monitor changes in register values at each breakpoint or step.

Peripheral View

You can access the Peripheral View by navigating to View > Peripheral View Panel. This view located on the panel, provides a comprehensive overview of the peripheral devices associated with the target system.

Click on a peripheral to expand its details and see its registers. Common peripherals inlcude timers, UARTs, GPIO ports, ADCs, and more. The view often presents a hierarchical tree structure showing the peripherals available on the microcontroller. The current values of the registers are displayed, allowing you to view the data in different formats (decimal, hexadecimal, binary).

When using the Peripheral View, you may notice that the values of some peripheral registers are hightlighted in yellow. This indicates that these values have recently changed, providing a quick visual cue to help you identify and track updates in the hardware’s status or configuration.

Disassembly View

This view allows users to view the assembly level instructions generated from the compiled source code. It provides a detailed view of how the high-level code translates into low-level machine instructions that are executed by the CPU.

Several fields are present in the Disassembly view by default:

  • The first column displays the memory address of each assembly instruction, indicating where in the memory the instruction is stored and will be executed.

  • The second column contains the names of functions, labels, or symbols related to specific points in the code, such as the start of a function or the location of a jump in a loop.

  • The third column shows the binary machine code representation of each assembly instruction. This is the machine-level code that the CPU will execute.

  • The fourth and fifth columns show the correspoding assembly instruction.

The first line of each instruction block contains the source code line that matches each assembly instruction.

How to use the Disassembly View

  1. Set Up Your Development Environment
  • Debug Configuration: Ensure your project is built in debug mode to include symbolic information (this allows the debugger to map source lines to assembly instructions).
  • Start Debugging: Run your application in debug mode and set breakpoints where you want to analyze the code.
  1. Navigate Between Views
  • Accessing Source Code: When you hit a breakpoint, your debugger will show the source code line where execution is paused.
  • Open Disassembly View: Use the commands by pressing commands Ctrl+Shift+P and typing Open Disassembly View to open the Disassembly View. Or you can find this by right-click on a function in the call stack and select “Open Disassembly View”.
  1. Analyze Corresponding Instructions
  • Step Through the Code: Use stepping commands (like Step Over or Step Into) to advance the execution line by line.
  • Observe Changes: As you step through the source code, observe the corresponding assembly instructions in the Disassembly View.

Memory View

Memory View displays the raw data stored at specific memory addresses in the system.

In this example, 44443444 is the hexadecimal representation stored at the memory address 0x40011004. 44 is the least significant byte, followed by 44, then 34, and 44 again. When reinterpreted in little-endian format, this becomes 0x00007683, matching the value shown in the Peripheral View.

RTOS View

The RTOS View provides a comprehensive interface for monitoring and managing Real-Time Operating System (RTOS) tasks and resources within your application.

  • Task monitoring: This view display list of all tasks currently running in the system. Each task is shown with important details, including:
    • Task name: The identifier for the task
    • State: The current status of the task (e.g., running, ready, blocked, suspended)
    • Priority: The priority level assigned to the task, influencing its execution order compared to other tasks.
    • Stack Usage: Information about the memory consumed by the task’s stack, crucial for avoiding stack overflows.
  • Heap: Displays information about dynamic memory allocation, including total size, used memory, and available space.

  • Queue: Shows the status of message queues used for inter-task communication. It indicates the current and maximum length of queues, item sizes, and waiting tasks, facilitating efficient data exchange between tasks.

  • Semaphore: Displays information about semaphores used for synchronization and resource management. It indicates the current value, type (binary, counting, mutex, or recursive mutex), and waiting tasks, helping to prevent race conditions and ensure safe access to shared resources.
